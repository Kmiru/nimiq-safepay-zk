# Nimiq SafePay ZK – Technical MVP Spec v1.0

## 1. Objetivo del MVP

Construir una mini app dentro de Nimiq Pay que permita:

1. Escanear o pegar un link/QR de pago.

2. Leer el destinatario, monto, concepto y datos de expiración.

3. Verificar que el pago no fue manipulado.

4. Generar una prueba ZK en el navegador usando NoirJS + Barretenberg.

5. Verificar la prueba contra un contrato verificador EVM usando window.ethereum.

6. Mostrar el estado ✅ ZK proof valid.

7. Enviar un pago NIM usando sendBasicTransactionWithData.

8. Adjuntar únicamente el intentHash en el campo data de la transacción NIM.

El MVP debe demostrar el flujo completo:

QR / Link → SafePay verification → ZK proof → EVM verifier → NIM payment with intentHash

## 2. Principios técnicos del MVP

### 2.1 Seguridad

SafePay ZK debe detectar cambios en:

recipient  
amount  
memo/concept hash  
expiration  
nonce  
intentHash

Si el QR/link fue modificado, la app debe mostrar:

🚨 Payment modified  
Do not pay


### 2.2 Privacidad

La app NO debe publicar datos sensibles en la blockchain.

El campo data de la transacción NIM debe contener solamente:

safepay:v1:\<intentHash\>

No se deben incluir:

nombres  
emails  
teléfonos  
mensajes privados  
direcciones externas  
notas personales  
datos de identidad  
historial de pagos

### 2.3 Simplicidad

El usuario solo debe ver un flujo sencillo:

Scan  
Verify  
ZK proof valid  
Pay  
Receipt

La complejidad ZK debe quedar oculta detrás de la UI.

## 5. Payment link / QR format

### 5.1 Encoding format

El QR debe contener un link SafePay propio:

https://safepay-zk.app/verify?p=\<base64url\_payload\>

Para demo local:

http://localhost:5173/verify?p=\<base64url\_payload\>

El parámetro p contiene un JSON canónico codificado en base64url.

## 6. SafePay payload schema

### 6.1 Human-readable payload

\{  
  "version": "safepay-zk-v1",  
  "type": "nim-payment-intent",  
  "network": "nimiq-testnet",  
  "recipient": "NQXX XXXX XXXX XXXX XXXX XXXX XXXX XXXX XXXX",  
  "amountLuna": 1000000,  
  "displayAmount": "10 NIM",  
  "memoPublic": "Demo payment",  
  "memoSecretHash": "0x...",  
  "expiresAt": 1780000000,  
  "nonce": "demo-uuid-or-random-id",  
  "intentHash": "0x...",  
  "nullifier": "0x..."  
\}

### 6.2 Field definitions

| Field | Type | Public in QR | Goes on Nimiq chain | Notes |
| - | -: | -: | -: | - |
| version | string | yes | no | Payload version |
| type | string | yes | no | Must be nim-payment-intent |
| network | string | yes | no | nimiq-testnet for MVP |
| recipient | string | yes | no | Nimiq address shown to user |
| amountLuna | number | yes | no | Amount in Luna |
| displayAmount | string | yes | no | UI-only |
| memoPublic | string | yes | no | Non-sensitive concept |
| memoSecretHash | hex string | yes | no | Hash of private memo/secret |
| expiresAt | number | yes | no | Unix timestamp |
| nonce | string | yes | no | Unique random ID |
| intentHash | hex string | yes | yes | Attached to NIM tx data |
| nullifier | hex string | yes | no in MVP | Local/demo replay protection |



## 7. Canonical payload rules

Before hashing or verifying, the app must normalize the payment intent.

### 7.1 Canonical object

The canonical object for intentHash must be:

\{  
  "version": "safepay-zk-v1",  
  "type": "nim-payment-intent",  
  "network": "nimiq-testnet",  
  "recipient": "NQXX XXXX XXXX XXXX XXXX XXXX XXXX XXXX XXXX",  
  "amountLuna": 1000000,  
  "memoSecretHash": "0x...",  
  "expiresAt": 1780000000,  
  "nonce": "demo-uuid-or-random-id"  
\}


Do not include these fields inside the hash preimage:

displayAmount  
memoPublic  
intentHash  
nullifier  
UI labels

### 7.2 Canonical JSON

Rules:

- Sort keys alphabetically.  
- Remove whitespace.  
- Use integer values for amountLuna and expiresAt.  
- Use lowercase hex strings.  
- Normalize Nimiq recipient address spacing before hashing.


## 8. intentHash definition

## 8.1 MVP definition

The intentHash must bind the payment intent to three security domains:

1. SafePay ZK app context  
2. Nimiq payment network  
3. EVM verifier network

Canonical definition:

intentHash = hash(  
  "SAFEPAY-ZK-V1",  
  nimiqNetworkId,  
  evmVerifierChainId,  
  version,  
  type,  
  network,  
  recipientField,  
  amountLuna,  
  memoSecretHashField,  
  expiresAt,  
  nonceField,  
  salt  
)


### Field meaning

| Field | Meaning |
| - | - |
| "SAFEPAY-ZK-V1" | Domain separator. Binds the intent to SafePay ZK v1. |
| nimiqNetworkId | Nimiq network where the payment will happen. Example: TestAlbatross, MainAlbatross, or DevAlbatross. |
| evmVerifierChainId | EVM chain ID where the verifier contract is deployed. Example: Sepolia 11155111. |
| version | SafePay payload version. Example: safepay-zk-v1. |
| type | Intent type. Example: nim-payment-intent. |
| network | Human-readable payment network label. Example: nimiq-testnet. |
| recipientField | Field representation of the Nimiq recipient address. |
| amountLuna | Payment amount in Luna. |
| memoSecretHashField | Field representation of the private memo hash. |
| expiresAt | Unix timestamp expiration. |
| nonceField | Unique payment intent nonce. |
| salt | Random salt used to make the intent unique and non-guessable. |

### Security explanation

The domain separator binds the intent to SafePay ZK v1.

The nimiqNetworkId binds the intent to the intended Nimiq payment network, preventing testnet/mainnet confusion.

The evmVerifierChainId binds the proof verification to the specific EVM network where the verifier contract is deployed, preventing cross-EVM replay or verifier substitution.

This avoids using one ambiguous chain\_id for two different security domains.

### 8.2 Field conversion

The UI can work with strings, but the circuit should work with field elements.

Recommended conversion layer:

recipient → recipientField  
memoSecretHash → memoSecretHashField  
nonce → nonceField  
salt → saltField

The exact conversion method must be pinned after testing the selected Noir/Barretenberg versions.


For the first circuit prototype, use simplified field inputs:

recipientField  
amountField  
memoSecretField  
expirationField  
nonceField  
saltField

This avoids early complexity with string parsing inside Noir.

## 9. nullifier definition

### 9.1 MVP definition

nullifier = hash(intentHash, secret)

The nullifier is used in the MVP for local/demo replay protection.

### 9.2 Important limitation

In the MVP, verification uses eth\_call.

That means:

eth\_call verifies proof validity  
eth\_call does not write state  
eth\_call does not permanently mark a nullifier as used

Therefore, true on-chain replay protection is NOT part of the MVP.

### 9.3 Future version

A future contract may expose:

mapping(bytes32 =\> bool) public usedNullifiers;  
  
function verifyAndRegister(bytes calldata proof, bytes32\[\] calldata publicInputs) external returns (bool);

That future version would require gas and user confirmation.

## 10. Noir circuit: payment\_intent\_match

### 10.1 Goal

Prove that the user knows the private values that produce a public intentHash.


### 10.2 Private inputs

domainSeparatorField  
nimiqNetworkIdField  
evmVerifierChainIdField  
versionField  
typeField  
networkField  
recipientField  
amountLuna  
memoSecretHashField  
expiresAt  
nonceField  
saltField  
secretField

### 10.3 Public inputs

intentHash  
nullifier

## 10.4 Circuit logic

The circuit must:

1. Validate basic range constraints.  
2. Hash the full intent context:  
   domainSeparator,  
   nimiqNetworkId,  
   evmVerifierChainId,  
   version,  
   type,  
   network,  
   recipientField,  
   amountLuna,  
   memoSecretHashField,  
   expiresAt,  
   nonceField,  
   salt.  
3. Assert that the result equals the public intentHash.  
4. Derive nullifier from intentHash and secretField.  
5. Assert that the derived nullifier equals the public nullifier.  
6. Return true if all constraints pass.


## 10.5 Conceptual Noir pseudocode

**fn** main(  
    domain\_separator: Field,  
    nimiq\_network\_id: Field,  
    evm\_verifier\_chain\_id: Field,  
    version\_field: Field,  
    type\_field: Field,  
    network\_field: Field,  
    recipient\_field: Field,  
    amount\_luna: Field,  
    memo\_secret\_hash\_field: Field,  
    expires\_at: Field,  
    nonce\_field: Field,  
    salt\_field: Field,  
    secret\_field: Field,  
    intent\_hash: **pub** Field,  
    nullifier: **pub** Field  
) \{  
    *// Range constraints*  
    assert(amount\_luna \< 18446744073709551616); *// 2^64*  
    assert(expires\_at \< 4294967296);            *// 2^32*  
  
    **let** computed\_intent\_hash = hash(\[  
        domain\_separator,  
        nimiq\_network\_id,  
        evm\_verifier\_chain\_id,  
        version\_field,  
        type\_field,  
        network\_field,  
        recipient\_field,  
        amount\_luna,  
        memo\_secret\_hash\_field,  
        expires\_at,  
        nonce\_field,  
        salt\_field  
    \]);  
  
    assert(computed\_intent\_hash == intent\_hash);  
  
    **let** computed\_nullifier = hash(\[  
        intent\_hash,  
        secret\_field  
    \]);  
  
    assert(computed\_nullifier == nullifier);  
\}

The exact hash() implementation must be selected according to the Noir version and proof backend.

For the first prototype, all string-like values should be converted to Field values outside the circuit. The circuit should not parse strings.

## 11. ZK proof generation flow

### 11.1 Browser-side generation

The app must generate the proof in the browser.

Flow:

1. Load compiled Noir artifact: payment\_intent\_match.json.  
2. Prepare private inputs from the scanned SafePay payload.  
3. Prepare public inputs: intentHash and nullifier.  
4. Execute Noir circuit.  
5. Generate proof with Barretenberg/bb.js.  
6. Keep private inputs in memory only.  
7. Send only proof + public inputs to the verifier call.

### 11.2 Privacy rule

Private inputs must not be:

stored in localStorage  
sent to server  
attached to the NIM transaction  
submitted to the EVM verifier contract  
printed in production logs

For demo logs, use redacted values only.

## 12. EVM verifier flow

### 12.1 Contract

The Solidity verifier is generated from the compiled Noir circuit using Barretenberg.

The verifier contract must be deployed to:

Sepolia or another supported EVM test network

### 12.2 Verification mode

MVP verification uses:

eth\_call


Reason:

- no gas  
- no on-chain state changes  
- fast demo  
- no additional transaction confirmation

### 12.3 Expected result

The frontend receives:

true

Then displays:

✅ ZK proof valid

If the verifier returns false or throws:

❌ ZK proof invalid

## 13. Nimiq Provider flow

### 13.1 Provider initialization

The app initializes the Nimiq Provider through the Mini App SDK.

Conceptual flow:

**import** \{ init \} **from** '@nimiq/mini-app-sdk'  
  
**const** nimiq = **await** init()

### 13.2 Network readiness check

Before payment:

1. Call isConsensusEstablished()  
2. If false, block payment and show:  
   "Nimiq network is not ready yet."  
3. Call getBlockNumber()  
4. Use block height as validityStartHeight or to prepare transaction validity.


### 13.3 Payment call

The payment must use:

sendBasicTransactionWithData

Data format:

safepay:v1:\<intentHash\>

Example:

safepay:v1:0xabc123...

The data field must not contain anything else.

### 13.4 Payment success

After payment, the app stores a local receipt:

\{  
  "txHash": "...",  
  "recipient": "NQ...",  
  "amountLuna": 1000000,  
  "intentHash": "0x...",  
  "timestamp": 1780000000,  
  "zkVerified": **true**  
\}

## 18. Security limitations of MVP

### 18.1 Tamper detection scope

The MVP detects manipulation of SafePay-generated payment payloads.

It does not prove that every arbitrary QR code on the internet is trustworthy.

### 18.2 No recipient reputation

The MVP does not prove that a recipient is trusted, verified or safe.

It only proves that the payment matches a specific private intent.

### 18.3 No permanent nullifier tracking

Because the MVP uses eth\_call, nullifier reuse is not permanently blocked on-chain.

Replay protection is local/demo-only.

### 18.4 No sensitive data in NIM data field

The NIM transaction must only include:

safepay:v1:\<intentHash\>
